# MarketFlow: техническое задание и этапы разработки

## 1. Назначение документа

Документ определяет последовательность разработки MarketFlow. Каждый следующий
этап начинается только после того, как предыдущий запускается, проверяется и
сохраняется отдельным Git-коммитом.

Главный принцип: технология подключается тогда, когда появляется задача,
которую она решает. Redis, RabbitMQ, многопоточность и AOP не добавляются ради
списка зависимостей.

## 2. Целевая архитектура

Первая версия — модульный монолит на Spring Boot:

```text
Client
  -> Spring MVC Controller
  -> Service
  -> Spring Data JPA Repository
  -> Hibernate/JPA
  -> PostgreSQL
```

Инфраструктурные компоненты:

```text
Redis     -> кэш и временные данные
RabbitMQ  -> асинхронные события
Flyway    -> миграции PostgreSQL
Actuator  -> health checks и метрики
Docker    -> единый воспроизводимый запуск
```

## 3. Базовые технические решения

- Java 21.
- Maven Wrapper.
- Актуальная стабильная версия Spring Boot, выбранная через Spring Initializr.
- REST API без отдельного frontend на первых этапах.
- PostgreSQL является основным источником данных.
- Redis не является единственным источником критически важных данных.
- Денежные значения хранятся в `BigDecimal`.
- Время хранится в UTC через `Instant` либо `OffsetDateTime`.
- Внешние DTO отделены от JPA Entity.
- Внедрение зависимостей выполняется через конструктор.
- Транзакционные границы находятся в Service.
- Структура базы изменяется только новыми миграциями Flyway.
- Секреты передаются через переменные окружения.
- Реальные платёжные и карточные данные не используются.

## 4. Предлагаемая структура пакетов

Рекомендуется группировка по функциональным модулям:

```text
com.erik.marketflow
|-- auth
|-- user
|-- seller
|-- catalog
|-- inventory
|-- cart
|-- order
|-- payment
|-- refund
|-- admin
|-- analytics
|-- audit
|-- notification
|-- config
`-- common
```

Внутри модуля могут находиться `controller`, `service`, `repository`, `entity`
и `dto`. `common` не должен превращаться в хранилище случайных классов.

## 5. Правило завершения каждого этапа

Этап считается завершённым, когда:

- проект компилируется;
- приложение запускается;
- миграции применяются на чистой базе;
- новый сценарий проверен тестами;
- API возвращает корректные HTTP-статусы;
- в Git отсутствуют пароли и секреты;
- изменения сохранены отдельным понятным коммитом;
- документация этапа обновлена.

# Этап 2. PostgreSQL и Flyway

## Почему Flyway подключается рано

Entity должны соответствовать контролируемой схеме базы. Если сначала позволить
Hibernate создавать таблицы, а Flyway добавить позже, появится две конкурирующие
системы управления схемой.

## Задачи

1. Создать локальную базу `marketflow`.
2. Настроить `spring.datasource.*` через environment variables.
3. Установить `spring.jpa.hibernate.ddl-auto=validate`.
4. Создать каталог:

```text
src/main/resources/db/migration
```

5. Создать `V1__create_users_and_roles.sql`.
6. Добавить таблицы `users`, `roles`, `user_roles`.
7. Добавить справочные роли без паролей пользователей.
8. Проверить `flyway_schema_history`.
9. Написать первый repository test с PostgreSQL.

## Критерий готовности

Приложение запускается на пустой базе, Flyway создаёт таблицы, Hibernate успешно
выполняет `validate`.


## Критерий готовности

Приложение корректно работает после очистки Redis, а повторный запрос быстрее

# Этап 12. RabbitMQ и событийная обработка

## Зависимость

```text
spring-boot-starter-amqp
```

## Цель

Отделить вторичные операции от основной транзакции заказа.

## События

```text
SellerApproved
OrderCreated
OrderPaid
OrderCancelled
OrderShipped
OrderDelivered
RefundCompleted
```

## Задачи

1. Настроить exchange, queues и routing keys.
2. Отправлять уведомления асинхронно.
3. Добавить retry с ограничением.
4. Создать dead-letter queue.
5. Сделать consumer идемпотентным.
6. Не отправлять событие до успешного commit бизнес-транзакции.
7. Реализовать простой transactional outbox:
   - бизнес-операция и `outbox_event` сохраняются одной транзакцией;
   - отдельный publisher отправляет события в RabbitMQ;
   - после подтверждения событие отмечается опубликованным.
8. Проверить повторную доставку одного сообщения.

## Критерий готовности

Повторное сообщение не создаёт повторное уведомление или финансовую операцию,
а временная недоступность RabbitMQ не ломает оплату заказа.

# Этап 15. Логирование, аудит, AOP и Actuator

## Цель

Сделать систему диагностируемой без утечки секретов.

## Задачи

1. Использовать SLF4J/Logback и `@Slf4j`.
2. Добавить `traceId` через MDC для каждого HTTP-запроса.
3. Настроить уровни `ERROR`, `WARN`, `INFO`, `DEBUG`.
4. Не логировать пароли, JWT, idempotency secrets и персональные данные.
5. Создать неизменяемый `AuditEvent` для критических действий.
6. При необходимости использовать Spring AOP для аудита или измерения времени,
   но не скрывать в аспекте бизнес-логику.
7. Настроить Actuator health, metrics и info.
8. Добавить custom health indicators для PostgreSQL, Redis и RabbitMQ.
9. Определить ротацию файловых логов для local-окружения.

## Критерий готовности

По `traceId` можно проследить обработку запроса, Actuator показывает состояние
зависимостей, а аудит отвечает кто изменил бизнес-данные.

## Уровни тестов

1. Unit tests Service через JUnit 5 и Mockito.
2. Controller tests через MockMvc.
3. Repository tests с настоящим PostgreSQL.
4. Security tests через `@WithMockUser` и реальные JWT-сценарии.
5. Integration tests через `@SpringBootTest` и Testcontainers.
6. Concurrency tests последнего товара и двойной оплаты.
7. Тесты миграций на пустой базе.
8. Тесты повторной доставки RabbitMQ.
9. Тесты недоступности Redis.

## Обязательные сценарии

- два покупателя заказывают последний товар;
- повторная оплата с одним idempotency key;
- ошибка внутри транзакции приводит к rollback;
- продавец не читает чужой заказ;
- аналитик не изменяет данные;
- просроченный резерв освобождается один раз;
- возврат нельзя выполнить дважды;
- повторное RabbitMQ-событие безопасно игнорируется.

## Критерий готовности

Все тесты запускаются одной Maven-командой и не требуют вручную настроенной
локальной базы.

# Этап 18. Docker, сборка и CI

## Цель

Запускать весь проект одинаково на любом компьютере.

## Задачи

1. Создать multi-stage `Dockerfile` для Spring Boot приложения.
2. Создать `compose.yml` со следующими сервисами:

```text
marketflow-app
postgres
redis
rabbitmq
```

3. Добавить volumes для постоянных данных.
4. Добавить healthcheck для каждого контейнера.
5. Не хранить пароли в `compose.yml`; использовать `.env` и `.env.example`.
6. Настроить порядок старта через health conditions и повторные подключения.
7. Проверить запуск на чистой машине:

```bash
docker compose up --build
```

8. Настроить GitHub Actions:
   - компиляция;
   - тесты;
   - проверка сборки JAR;
   - сборка Docker image.
9. Создать команды остановки и очистки только для dev-окружения.

## Критерий готовности

health endpoint возвращает `UP`.

## Финальный сценарий приёмки

1. Покупатель регистрируется.
2. Пользователь подаёт заявку продавца.
3. Модератор одобряет заявку.
4. Продавец создаёт товар и пополняет остаток.
5. Два покупателя одновременно пытаются купить последний экземпляр.
6. Только один заказ получает резерв.
7. Покупатель дважды отправляет запрос оплаты с одним ключом.
8. Деньги списываются один раз.
9. Комиссия начисляется платформе, доход — продавцу.
10. Продавец собирает и отправляет заказ.
11. Покупатель подтверждает доставку.
12. Аналитик видит операцию в отчёте.
13. Владелец видит баланс, сотрудника и запись аудита.
14. Все сервисы запускаются через Docker Compose.
15. CI автоматически выполняет тесты.
